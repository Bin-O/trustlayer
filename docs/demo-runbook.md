## §3. データ復位手順

デモ/リハで状態を変えた場合の戻し方。**書込を伴うため実行前に内容を確認**。

### §3-A. リハで面談を保存した後 → seed 全再投入
`seed_arch_demo.sql` は冒頭で当組織のデモ8名を削除（`foreign_workers` の
FK CASCADE で support_records / support_tasks / document_generations /
qualifications / worker_work_assignments / trust_score_snapshots も一括削除）→
再投入する。面談保存で増えた記録・様式もこれで初期状態に戻る。

```bash
# .env.local の SUPABASE_PAT を使い Management API 経由で実行
node run_sql.mjs seed_arch_demo.sql
```

確認ポイント（実行後）:
- 出力が「エラー 0」。
- `/dashboard`: ⑥サリ・デウィの四半期面談 Q2 が **超過（赤）** に復活、超過2。
- `/reports/support-matrix`: 全体 **6/42**、⑥に ⚠️、③ **4/6**。
- `/employees/…0003`: 信頼スコア **94/100**。

### §3-B. ⑥に作業を追加する実演をした後 → 個別復位
⑥（サリ・デウィ）に UI で `forklift_1t` を追加すると橙119条カードが出る。
実演後は下記で削除する。

```sql
DELETE FROM worker_work_assignments
  WHERE worker_id = 'a1000000-0000-4000-8000-000000000006'
    AND work_key = 'forklift_1t';
```

⚠️ **重要**: `worker_work_assignments` には anon/authenticated の DELETE ポリシーが
無いため、UI や anon SQL からの直接 DELETE は **0行＝無音失敗**する。必ず
`run_sql.mjs`（Management API＝service role）経由で実行すること。

実行手順（一時ファイル経由。run_sql.mjs はSQL文字列を直接受け取れずファイル指定のため）:
```bash
# プロジェクト直下に一時 .sql を作成して実行し、後で削除
printf "DELETE FROM worker_work_assignments WHERE worker_id='a1000000-0000-4000-8000-000000000006' AND work_key='forklift_1t';\n" > _restore.sql
node run_sql.mjs _restore.sql
rm -f _restore.sql
```
（§3-A の全再投入を行う場合は CASCADE で消えるため §3-B は不要。）

確認ポイント: `/employees/…0006` で作業資格チェックの橙カードが消え、
「＋ 作業を追加」で forklift_1t が再び選択肢に戻っている。

## §5. 想定問答（未実装部分への口頭応対）

Q1. **料金はいくら？**
> 現在は機能検証段階で定価は未確定。従業員数ベースの月額SaaS課金（自社支援が現実的な
> 10〜100名規模の中小企業がターゲット）を想定しています。

Q2. **複数企業（マルチテナント）で使える？**
> 現版は単一テナント設計で、マルチテナント化（組織ID・担当者IDによる行レベル権限＝RLS、
> API層の認可）は設計済みで拡張時に実装します。まず 10〜100名規模で成功モデルを固め、
> 標準化してから拡張する方針です（product-direction.md にロードマップ記載）。

Q3. **外国人本人向けのアプリは？**
> 現版は受入企業・支援担当者向けです。本人向け（多言語通知・面談同意・相談窓口）は
> Phase 3 のビジョンで、行政書士法改正の「自社作成支援ソフト」定位を厳守して設計します。

Q4. **健診など他の義務にも対応する？**
> 雇入時健診・定期健診・特定業務健診・特別教育等は、既存のタスクエンジンに one_time /
> recurring として載せる設計が済んでおり Phase B で実装します（現状は支援・義務フローに
> 「対応予定」の破線表示のみ）。作業割当をトリガーに資格・健診・教育を判定する仕組みは、
> フォークリフト資格判定と同一メカニズムで一般化します。

Q5. **対応業界を増やすには？**
> 業界パッケージ（`lib/industry/packages/*.ts`）にタスク・資格ルール・帳票を**データとして
> 追加するだけ**です。業界専用コードは書かない設計なので、現在の運送・製造から特定技能14分野へ
> 段階的に拡張できます。
