# プロダクト方向性(2026-07-11 決定)

## 定位
特定技能を知らない総務担当者でも、監査に耐える自社支援ができる「ナビゲーター型」SaaS。

## 設計原則(全機能に優先適用)

### 原則1:同じ情報を二度入力させない
- 一度入力された情報は、書類生成・タスク記録・届出・評価の全場面で自動引用する
- 既存の実践:在留カードAI読取、organization_defaults プリフィル、
  雇用条件変更→随時届出27項目diff自動抽出
- 新機能への制約:
  - 面談フォームは既存情報を全て自動表示し、新規増分のみ入力させる
  - 法定文書はフォーム保存と同時に生成する(二度書き工程を作らない)
  - 紙・画像情報の取込は Vision AI 路線を優先検討する

### 原則2:情報は一箇所に集約、遺漏を構造的に不可能にする
- 三層の集約構造:
  - 個人層:従業員詳細=その人の全て
  - 会社層:ダッシュボード「今やること」=全員の全期限
  - 証拠層:document_generations + support_records=監査対応の底帳
- 新機能への制約:全てのタスク・期限は必ず「今やること」に自動集約される。
  子ページにしか存在しない期限を作ることを禁止。業界パッケージ追加時も
  タスクエンジン経由で自動継承する

### 原則3:乱れず、しかし全部見せる(2026-07-16 決定・全改修の判断基準)
- 情報は削らない。義務・状態・欠缺はすべて画面に残す(監査ツールの本分)。
- 乱れの解消は削除ではなく以下の3手段で行う:
  1. **階層**:各画面に主役は1つ。ダッシュボードの主役は赤い警報。
     統計カードや期日前の緑タスクは視覚的に一段引く(サイズ/彩度/位置)。
  2. **動線**:情報は雇用ライフサイクルの工程順に並べる(機能種別順ではなく)。
  3. **一貫**:同じ意味は全画面で同じ色・同じ形。
     赤=期限超過、または回復不能な期限の切迫(在留期限 残30日以内) /
     橙=検証された欠缺・要対応 / 灰=未蓄積・期日前 / 緑=完了・検証済。
     この4色の意味体系に反する色使いは監査で全て洗い出す。
  - 赤の例外規定(2026-07-16 裁定):在留期限は超過=不法滞在で回復不能なため、
    「超過してから赤」では警報として遅い。在留期限に限り ≤30日=赤 /
    31〜60日=橙 / 61〜90日=灰 とする。通常のタスク期日は超過のみ赤。
- UI監査時の必須3観点:各画面について「主役は何か/工程順になっているか/
  4色体系に反する箇所はどこか」を点検する。

## 5つの柱

### 1. ナビゲーター設計
従業員状態駆動のタスクフロー(今やること)が総務のデフォルト画面。
各タスクは三段式(これは何?/どうやる?/記録する)。
10支援業務マトリクスはレポートビュー(監査・投資家向け)に降格。

### 2. 物理業務への価値
ソフトは事を代行しない。「知らない」と「証拠がない」を消す:
- 事前:チェックリスト+LINE提醒
- 事中:多言語テンプレート
- 事後:ワンクリック記録→法定文書自動生成

### 3. 評価は面談の副産物
Phase 2 は独立ページではなく、タスクフロー初の周期タスクとして実装する。
四半期面談タスク→三段式→末尾に雇用主評価→evaluations→
信頼スコア面談時評価15点、で閉環。

### 4. 業界精細化=タスクエンジン(共通)+業界設定パッケージ(データ)
追加タスクチェーン・アラートルール・文案・帳票を業界パッケージとして表現し、
業界専用コードは書かない。運送業の特定活動55号→免許→特技1号の期限
カウントダウン(トラック6ヶ月/バス・タクシー1年)は業界パッケージの
タスクチェーンの一例。
- **Phase B TODO(原則2完全準拠)**: 段3の作業資格ギャップ(労安衛法119条リスク)は
  現状 従業員詳細ページのみに表示。ダッシュボード「今やること」への集約は
  非日付の「リスク枠」として Phase B で設計する(状態リスクの urgency モデルを含む)。

### 5. 専門家連携
デモ前はタスク指引末尾の静的情報リンク(行政書士会等の公的検索ページ)のみ。
マッチングは Phase 3 ビジョン(行政書士法改正の「自社作成支援ソフト」定位を厳守)。

## デモ前実施順序
1. Phase 2 面談タスク様板
2. 10業務マトリクス
3. 業界フィールド+運送業チェックリスト
4. デモデータ+UI改修+リハーサル

## スケーラビリティの考え方(2026-07-13 決定)

### 現在の最適規模
- ターゲットは自社支援が現実的な中小企業。特定技能外国人 10〜100 名規模に最適化する。
- 1000 名規模の企業は通常、専門の人事部門を持つか登録支援機関に委託しており、
  当面のターゲットではない。

### 1000名規模で必要になる3つの変更(設計上すでに想定内)

1. **担当者ごとのタスク分割(最重要)**
   - 現在の「今やること」は会社全体の全期限を1画面に集約する設計。1000 名規模では
     総務が1人ではなく、部門・拠点・支援担当者ごとに分業する。
   - ダッシュボードの主語を「会社の全タスク」から「私の担当分のタスク」へ転換し、
     管理者は全体ビューに切替可能とする。
   - 支援責任者・支援担当者フィールドは法定要件として既に保持しており、この分割軸は
     最初から設計に内在している(=規模拡大時の設計変更ではない)。

2. **タスク生成のバッチ化**
   - 現在はレイジー生成(ダッシュボード読込時に当四半期分を upsert)。1000 名 × 2種
     = 2000 件の upsert はページ読込を圧迫するため、cron(pg_cron 等)によるバッチ
     生成へ切り替える。
   - 信頼スコアのスナップショット生成も同様。

3. **RLSによる権限分離**
   - 現在は anon 全許可ポリシーの単一テナント設計(開発段階の既知債務)。
   - マルチテナント化+行レベル権限(組織ID・担当者IDによる絞り込み)が必須。
   - API層の認可も同時に実装する(現在 /api/* はガード対象外)。

### UI層の対応(比較的軽微)
- 従業員一覧・10業務マトリクスのページング/仮想スクロール/絞り込み
  (部門・分野・状態)/検索。
- 従業員一覧の信頼スコアバッジ(緑検証済/橙要対応/灰蓄積中)は現在、行ごとに
  フル `calculateTrustScore`(1人=約7クエリ)を並列実行している。1000名規模では
  分岐判定専用の軽量クエリ化、または `trust_score_snapshots`(バッチ生成済み)の
  参照に切り替える。

### 方針
「境界がどこにあるかを知り、突破方法も設計済みだが、今は意図的に作らない」。
まず 10〜100 名規模で成功モデルを構築し、標準化してから拡張する
(業界特化戦略と同じ考え方)。
